# Git 多版本树导出工具 — 设计说明（v1.2）

本文描述 **git-export-merge** 的设计目标、数据模型、算法语义、配置约定及**兼容层**的定位。实现以仓库内 `src/git_export_merge_two_commits.py` 为准。

---

## 一、背景与要解决的问题

在交付、验收或归档场景中，常需要把**多个 Git 提交**（同一仓库上的不同快照）里的文件放到一个普通目录里，便于：

- 与非 Git 工具链对接；
- 人工比对或打包外发；
- 保留「同一相对路径上多个不同内容版本」而不丢失。

本工具**不创建**新的 Git 合并提交，只做**文件树层面的导出**。

---

## 二、术语

| 术语 | 含义 |
|------|------|
| **提交（commit）** | 通过 `rev-parse …^{commit}` 解析得到的一个快照根。 |
| **路径** | `git ls-tree -r` 得到的相对路径（POSIX 风格 `/`）。 |
| **blob / oid** | 某路径下文件内容在 Git 中的对象 id；同一 oid 即字节级相同。 |
| **commits 列表** | 用户指定的、参与本次导出的提交序列，顺序有意义（见第四节「标签」）。 |

---

## 三、功能语义（核心算法）

### 3.1 输入

- 解析后的提交列表：`C₁, C₂, …, Cₙ`（`n ≥ 1`）。
- 对每个 `Cᵢ` 读取完整文件树：`path → (mode, type, oid)`，仅处理 **blob**（普通文件内容）。

### 3.2 路径集合

取所有提交中出现过的路径的**并集**，记为 `P`。

### 3.3 对每个路径 `p ∈ P` 的决策

1. 收集每个包含 `p` 的提交上的 `(commit, oid)`（跳过不含 `p` 的提交；非 blob 按策略报错或跳过）。
2. 按 **oid 去重**，并保持「**首次出现顺序**」：即按 `C₁…Cₙ` 扫描，每个新 oid 第一次出现时记下其**代表提交**，用于生成文件名标签。
3. **若仅一种 oid**：导出**一个**文件，相对路径仍为 `p`。
4. **若多种 oid**：为每种 oid 各导出一份；路径改为在文件名主干与扩展名之间插入 `.__<标签>__`，其中 `<标签>` 为代表提交的 **SHA 前缀**，长度在 `tag_min_len`～`tag_max_len` 间尝试；若仍与同一 `p` 下其它变体冲突，则自动加长直至使用完整哈希或加后缀，保证**同一原路径下输出文件名唯一**。

### 3.4 非目标（明确不做）

- 不做语义合并、不做冲突解决策略（始终以 blob 为粒度）。
- 不展开 submodule（`gitlink`）；默认遇非 blob 失败，可用 `--skip-non-blob` 跳过并记录。

---

## 四、配置与命令行

### 4.1 优先级

1. 命令行中**显式出现**的选项，覆盖配置文件中的同名字段。  
2. 配置文件路径：`--config` / `--config=`，或环境变量 `GIT_EXPORT_MERGE_CONFIG`。  
3. 配置文件内：`repo`、`output` 若为相对路径，则相对于**配置文件所在目录**解析。

### 4.2 推荐配置形态

- 使用仓库根目录的 `config.example.json` 为模板，复制后改名为自用文件（勿提交含密钥或内网路径的私用配置到公共仓库）。
- 主字段：`commits`（字符串数组）、`repo`、`output`，以及 `tag_min_len` / `tag_max_len`、`dry_run` 等。

---

## 五、兼容策略（专门讨论）

### 5.1 为何存在兼容层

工具在演进过程中曾出现「`commit_first` + `commit_second`」或命令行 `--commit-first` / `--commit-second`」等形态；当前主模型已统一为 **`commits` 单列表 + 路径并集语义**。  
兼容层的目的只有一条：**让已有配置文件或脚本在不大改的前提下仍能跑起来**，降低一次性迁移成本。

### 5.2 配置文件兼容（建议保留，但应迁移）

| 情况 | 行为 |
|------|------|
| 存在 `commits` | **仅使用** `commits`，忽略 `commit_first` / `commit_second`（若同时写了旧键，以 `commits` 为准，避免歧义）。 |
| 无 `commits`，但有 `commit_first` 和/或 `commit_second` | 按顺序拼成：`[…expand(commit_first)…, commit_second（若有）]`，再进入第三节算法。 |

**重要语义说明**：旧名「first / second」容易让人联想到「先合并一侧再与另一侧比较」的两阶段模型；**当前实现没有该模型**，只有「列表中所有提交参与并集」。若旧配置依赖的是早期脚本里**不同的**合并语义，仅靠自动拼接**不一定**等价；迁移时应以第三节为准，人工核对 `commits` 顺序与是否覆盖所需版本。

**文档结论**：兼容块**保留在代码中**，但在本设计说明中明确为 **过渡机制**；**新配置一律只写 `commits`**。

### 5.3 命令行兼容（建议标弃用，可考虑未来大版本移除）

| 情况 | 行为 |
|------|------|
| 使用 `--commit`（可多次） | 推荐；语义清晰。 |
| 使用 `--commit-first` / `--commit-second` | 仍可解析为 `commits` 列表；运行时会输出**弃用警告**，提醒改用 `--commit`。 |

**是否去掉**：  
- **短期内**：建议**保留**，避免破坏已存在的 CI/个人脚本。  
- **中长期**：在 README/变更日志中声明「将在未来主版本移除」；当确认无外部依赖后，可在大版本（如 2.0）删除 CLI 旧参数，**仅保留**配置文件里对旧键的读取一段过渡期，再完全删除。

### 5.4 `hash_prefix_len` 与 `tag_min_len`

历史配置或命令行曾使用 `hash_prefix_len`。当前以 `tag_min_len` / `tag_max_len` 表达「冲突标签长度范围」。  
实现上：`hash_prefix_len` 可映射为 `tag_min_len`（仅当未显式指定 `tag_min_len` 时），属于**窄兼容**，建议在配置中改为新键名。

### 5.5 小结表

| 兼容项 | 建议 |
|--------|------|
| 配置 `commits` | **唯一推荐** |
| 配置 `commit_first` + `commit_second` | **过渡用**，尽快改为 `commits` |
| CLI `--commit-first` / `--commit-second` | **不推荐**；保留至大版本再评估删除 |
| `hash_prefix_len` | 映射到 `tag_min_len`；配置应迁移 |

## 六、实现要点（与 Git 的交互）

- 树遍历：`git ls-tree -r -z <commit>`。  
- 读内容：`git cat-file --batch`；注意每条 blob 记录后的**换行分隔符**不属于文件内容，解析时必须跳过。  
- 安全：导出根目录不得落在目标 `repo` 根或其 `.git` 之下，避免误删仓库。

---

## 七、风险与限制

- 大仓库、大量二进制：时间与磁盘占用高；务必先用 `dry_run` 看统计。  
- 标签碰撞：由程序自动加长哈希前缀处理；极端情况下文件名会较长。  
- 兼容层误用：误以为「first+second」仍代表旧的两阶段语义时，结果可能与直觉不符——**以第三节并集算法为唯一权威**。

---

## 八、版本与变更

设计语义以 `src/git_export_merge_two_commits.py` 中的 `__version__` 及仓库提交历史为准；重大语义调整应同步更新本节与 README。
