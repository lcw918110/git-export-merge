# Git 多版本树导出工具 — 设计说明

## 1. 目标

在**不依赖合并提交对象**的前提下，把多个 Git 提交（快照）中的文件以**目录并集**方式导出到普通文件夹，供交付、比对或归档。

## 2. 核心语义

- **输入**：有序列表 `commits`，每一项解析为一个 **commit**（完整对象名）。
- **路径并集**：取所有提交中出现过的相对路径的并集。
- **同一路径**：
  - 若各提交在该路径上的 **blob oid 相同**（内容一致），则导出 **一个** 文件，路径保持原相对路径。
  - 若 **oid 不同**，则为每个不同内容各导出一份；在文件名「主干」与「扩展名」之间插入 `.__<标签>__`，避免覆盖。
- **标签**：默认取「该 oid **首次**在列表顺序中出现」时对应提交的 **SHA 前缀**；长度在 `tag_min_len`～`tag_max_len` 间自动调整，确保同一原路径下标签不冲突，必要时使用完整哈希或加后缀。

## 3. 非目标

- 不做三路合并、不做语义合并；仅按 **blob 字节级**区分版本。
- 不处理 submodule（`gitlink`）内容展开；默认遇到非 blob 即失败，可用 `--skip-non-blob` 跳过并记录。

## 4. 配置与 CLI 优先级

1. 命令行**显式出现**的参数覆盖配置文件同名字段。
2. 配置文件路径：`--config` / `--config=`，或环境变量 `GIT_EXPORT_MERGE_CONFIG`。
3. 配置文件中的相对路径 `repo`、`output`：相对于**配置文件所在目录**解析。仓库根目录提供 `config.example.json` 供复制。

## 5. 兼容策略

历史配置若仍使用 `commit_first`（字符串或数组）+ `commit_second`，会按顺序拼成 `commits` 列表。命令行仍接受 `--commit-first` / `--commit-second`，但会提示弃用，建议统一为 `--commit`。

## 6. 实现要点

- 使用 `git ls-tree -r -z` 构建各提交的 path→(mode, kind, oid)。
- 使用 `git cat-file --batch` 批量读取 blob；注意 batch 输出在每条 blob 后的换行分隔符，避免把分隔符读进内容。
- 导出前校验输出目录不得落在目标仓库根或 `.git` 下，防止误删仓库。

## 7. 风险与限制

- 超大仓库/大量二进制时导出耗时与磁盘占用高；建议先 `--dry-run` 看统计。
- 极罕见情况下不同提交的前缀在设定长度内碰撞，脚本会自动加长或降级为完整哈希。
